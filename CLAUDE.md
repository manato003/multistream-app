## ワークフロー

### 1. 計画優先
- 3ステップ以上 or アーキテクチャに関わる変更は必ず実装前に計画を言語化する
- 途中でおかしくなったら無理に進めず即停止して再計画
- 曖昧な仕様は実装前にユーザーに確認する

### 2. 自己改善ループ
- ユーザーから修正を受けたら `tasks/lessons.md` にパターンとして記録する
- 同じミスを繰り返さないためのルールを自分で書く
- セッション開始時に `tasks/lessons.md` を確認する

### 3. 完了前の検証
- 「動いた」で終わりにしない。TypeScriptエラーゼロ・Viteエラーゼロを確認してから完了とする
- 自分に問う:「この実装はシニアエンジニアが承認するか？」

### 4. エレガントさの追求（バランス重視）
- 非自明な変更をする前に「もっとエレガントな方法はないか？」と一度立ち止まる
- ハック的な修正になりそうなら「知っている全てを踏まえた上でエレガントな解を実装する」
- 明らかにシンプルな修正には過剰設計しない

### 5. バグは自律的に修正
- バグ報告を受けたらログ・エラー・症状から根本原因を特定して修正する
- ユーザーに手順を聞かない。自分で調べて直す

## タスク管理

1. **計画を書く**: 実装前に `tasks/todo.md` にチェックリスト形式で計画を書く
2. **確認する**: 実装開始前にユーザーに計画を共有する
3. **進捗を追う**: 完了したものをチェックしながら進める
4. **変更を説明する**: 各ステップで何をしたか高レベルのサマリーを伝える
5. **結果を記録する**: `tasks/todo.md` にレビューセクションを追加する
6. **学びを残す**: 修正を受けたら `tasks/lessons.md` を更新する

---

## コア原則

- **シンプルさ優先**: 変更は最小限に。影響範囲を絞る。
- **手を抜かない**: 一時的な修正をしない。根本原因を探す。シニア開発者の基準で。
- **影響を最小化**: 必要な箇所だけ触る。不要なバグを持ち込まない。

## プロジェクト固有のルール

- コードを編集したら**会話の最後に必ずgit commit**する
- コミット: `cd "C:\Users\manat\OneDrive\Desktop\claude\multistream-app"; git add .; git commit -m "メッセージ"`
- iframeを扱う変更は特にテストを慎重に（HTML5 DnD非対応、elementFromPoint方式を使用中）
- `StreamFrame` の `key` に `reloadKey` を含めない（全フレームリマウントのバグが再発する）
- テーマはダークモードのみ（ライトモード復活させない）
